## Section 1: `## RBAC — How Role-Based Access Works (Dual Enforcement)`

1. Subsection `### Why Two Layers?`:
   - Explain that RBAC is enforced in TWO places for defense-in-depth:
     a. **Client-side** (browser): hides nav items + blocks routes — fast UX, no unnecessary API calls
     b. **Server-side** (PHP): returns 401/403 on every endpoint — the real security gate
   - If someone bypasses the client (e.g., types URL directly), the server still blocks them

2. Subsection `### Layer 1: Client-Side (What the User Sees)`:
   - Explain `shell.html` nav items have `data-roles="admin,program_head"` attributes
   - `router.js` `applyRbacToNav()` reads user role, hides any nav item whose data-roles doesn't include it
   - `router.js` route definitions have `roles: ['admin', 'program_head']` — if user role not in array, redirect to dashboard
   - Result: Program Head never sees "Users" or "Faculty" in sidebar, and can't navigate to those routes

3. Subsection `### Layer 2: Server-Side (The Real Guard)`:
   - Explain every PHP endpoint starts with:
     a. Check `$_SESSION['user_id']` — if empty, return `401 Not authenticated`
     b. Check `$_SESSION['role']` against allowed list — if not allowed, return `403 Forbidden`
   - This means even if someone manually calls `api/users/index.php`, they get 403 unless they're admin
   - The PHP role checks exactly match the RBAC matrix in REQUIREMENTS.md

4. Subsection `### RBAC Matrix (Who Can Do What)`:
