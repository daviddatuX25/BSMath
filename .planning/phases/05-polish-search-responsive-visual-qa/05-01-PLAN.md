# Plan 05-01: Global Search + Responsive Shell

## Context

Phase 5 is the final polish phase. The SPA is fully functional with auth, RBAC, and all CRUD modules. This plan adds two UX features that the rubric requires:

1. **Global search** (R14) — the topbar already has a `#topbar-search` input, but it does nothing. We wire it up.
2. **Responsive layout** (R15) — the sidebar is `w-64 fixed` and never collapses. Below `md` (768px), it must become a slide-out drawer.

Both are CSS/JS-only changes — no PHP, no new API endpoints.

- assets/js/views/programs.js (for list rendering pattern)
- assets/js/router.js (for NAV_ITEMS)

<action>
Create a new file `assets/js/ui/search.js` with these exports:

```js
// assets/js/ui/search.js
// Debounced global search — filters rows in the currently-mounted list view.

const DEBOUNCE_MS = 250;
let currentView = null; // set by router on each navigation

/**
 * Call this when the search input receives input.
 * Debounces by DEBOUNCE_MS, then filters visible rows.
 */
export function initSearch() {
  const input = document.getElementById('topbar-search');
  if (!input) return;
  let timer;
  input.addEventListener('input', () => {
    clearTimeout(timer);
    timer = setTimeout(() => filterCurrentView(input.value.trim()), DEBOUNCE_MS);
  });
}

/**
 * Set which view module is currently mounted.
 * Called by router.js after each navigation.
 * @param {object} view — { name: string, filterRows: (term) => void }
 */
export function setActiveView(view) {
  currentView = view;
  // Clear search when navigating away
  const input = document.getElementById('topbar-search');
  if (input) input.value = '';
}

/**
 * Filter the current view using the provided search term.
 * Falls back to a simple text-content filter if the view
 * does not expose filterRows().
 */
function filterCurrentView(term) {
  if (!currentView) return;
  const lower = term.toLowerCase();
  // View-provided filter (preferred — module-specific column matching)
  if (typeof currentView.filterRows === 'function') {
    currentView.filterRows(lower);
    return;
  }
  // Generic fallback: hide/show table rows or card elements that don't match
  const main = document.getElementById('main-content');
  if (!main) return;
  const rows = main.querySelectorAll('tr[data-id], .searchable-card');
  rows.forEach(row => {
    const text = row.textContent.toLowerCase();
    row.style.display = text.includes(lower) ? '' : 'none';
  });
}
```

Key points:
- `DEBOUNCE_MS` is 250 per R14.
- Each view module (dashboard, programs, etc.) can optionally expose `filterRows(term)` for column-aware filtering. If not, the generic fallback hides/shows rows by text content.
- `setActiveView` is called by the router on every navigation so the search module knows which view is active.
- On navigation, the search input is cleared so stale queries don't persist.
</action>

<acceptance_criteria>
- `assets/js/ui/search.js` exists and exports `initSearch` and `setActiveView`
- File contains `DEBOUNCE_MS = 250`
- `filterCurrentView` calls `currentView.filterRows(lower)` when available
- Generic fallback queries `tr[data-id]` or `.searchable-card` elements
</acceptance_criteria>

<action>

### A. Export `filterRows` function

Each view module already exports a `loadXxx(user)` function. Add a named export `filterRows` that takes a lowercase search term and hides/shows the relevant rows.

**Dashboard** — no table rows to filter, but the stat cards should be excluded from search. Skip `filterRows` for dashboard (or return early).

**CRUD views (programs, announcements, events, news, gallery, faculty, users, approvals)** — each renders a `<table>` with `<tr data-id="...">` rows. The `filterRows` function should:

```js
export function filterRows(term) {
  const rows = document.querySelectorAll('#main-content tr[data-id]');
  if (!term) {
    rows.forEach(r => r.style.display = '');
    return;
  }
  rows.forEach(r => {
    r.style.display = r.textContent.toLowerCase().includes(term) ? '' : 'none';
  });
}
```

### B. Register as active view in router

In `router.js`, after each view's loader runs, call `setActiveView` with an object that has the view's `filterRows`. For example, the programs loader becomes:

```js
'/programs': {
  roles: ['admin', 'program_head'],
  loader: async (user) => {
    await ensureShell();
    highlightNav('/programs');
    const mod = await import('./views/programs.js');
    const { loadPrograms, filterRows } = mod;
    await loadPrograms(user);
    setActiveView({ name: 'programs', filterRows });
  },
},
```

Do this for ALL list views: programs, announcements, events, news, gallery, faculty, users, approvals. Dashboard can skip `filterRows` or export a no-op.

Import `setActiveView` from `./ui/search.js` at the top of router.js.
</action>

## Task 6: Add hamburger toggle logic to app.js

<read_first>
- assets/js/app.js
- views/shell.html
</read_first>

<action>
In `assets/js/app.js`, add a `setupMobileMenu()` function and call it from the existing init flow. This function:

1. Finds `#hamburger-btn` and `#sidebar-backdrop`
2. Toggles `.sidebar-open` on both `#sidebar` and `#sidebar-backdrop` when hamburger is clicked
3. Closes the drawer when backdrop is clicked or when a nav link is clicked on mobile
4. Sets `aria-expanded` on the hamburger button

Add after the existing imports:

```js
/** Toggle the mobile sidebar drawer. */
function setupMobileMenu() {
  const btn = document.getElementById('hamburger-btn');
  const sidebar = document.getElementById('sidebar');
  const backdrop = document.getElementById('sidebar-backdrop');

  if (!btn || !sidebar || !backdrop) return;

  const open = () => {
    sidebar.classList.add('sidebar-open');
    backdrop.classList.add('sidebar-open');
    btn.setAttribute('aria-expanded', 'true');
  };
  const close = () => {
    sidebar.classList.remove('sidebar-open');
    backdrop.classList.remove('sidebar-open');
    btn.setAttribute('aria-expanded', 'false');
  };

  btn.addEventListener('click', () => {
    const isOpen = sidebar.classList.contains('sidebar-open');
    isOpen ? close() : open();
  });

  backdrop.addEventListener('click', close);

  // Close drawer when a nav link is clicked on mobile
  sidebar.querySelectorAll('.nav-item').forEach(link => {
    link.addEventListener('click', () => {
      if (window.innerWidth < 768) close();
    });
  });
}
```

Then call `setupMobileMenu()` right after `router.init()` — this is safe because `ensureShell()` (which injects shell.html) runs on first navigation, and `setupMobileMenu()` only needs to find elements that exist after shell mount. However, `setupMobileMenu()` must run AFTER the shell is mounted. The cleanest approach:

In `router.js`, add a call to `setupMobileMenu()` inside the `ensureShell()` function, right after `applyRbacToNav()` and `populateSidebarUser()`. Export `setupMobileMenu` from `app.js` and import it in `router.js`.

Actually, even simpler: add the setup call inside `ensureShell()` in router.js:

```js
// Inside ensureShell(), after populateSidebarUser():
const { setupMobileMenu } = await import('./app.js');
setupMobileMenu();
```

But this creates a circular dependency (app imports router, router imports app). Instead, use a simpler approach:

**In router.js, at the end of `ensureShell()`, dispatch a custom event:**

```js
window.dispatchEvent(new CustomEvent('shell-ready'));
```

**In app.js, listen for it:**

```js
window.addEventListener('shell-ready', setupMobileMenu);
```

This avoids circular imports entirely.
</action>

<acceptance_criteria>
- Clicking `#hamburger-btn` toggles `.sidebar-open` on `#sidebar` and `#sidebar-backdrop`
- Clicking the backdrop closes the drawer
- Clicking a nav item on mobile (< 768px) closes the drawer
- `hamburger-btn` has `aria-expanded` attribute that toggles
- No circular import between app.js and router.js
- `router.js` dispatches `shell-ready` event at end of `ensureShell()`
