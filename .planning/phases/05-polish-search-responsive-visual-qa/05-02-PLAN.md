```js
// assets/js/ui/empty-state.js
// Reusable empty-state component for list views.
// Shows an icon, title, and optional subtitle when a list has zero items.

/**
 * Returns HTML for an empty-state placeholder.
 * @param {string} icon — Material Symbol name (e.g. 'inbox', 'event')
 * @param {string} title — Primary message (e.g. 'No programs yet')
 * @param {string} [subtitle] — Secondary message (e.g. 'Create your first program to get started.')
 * @returns {string} HTML string
 */
export function emptyState(icon, title, subtitle = '') {
  return `
    <div class="flex flex-col items-center justify-center py-16 text-center">
      <div class="w-16 h-16 bg-stone-100 rounded-2xl flex items-center justify-center mb-4">
        <span class="material-symbols-outlined text-stone-400 text-[32px]">${icon}</span>
      </div>
      <p class="text-stone-700 font-semibold text-lg">${title}</p>
      ${subtitle ? `<p class="text-stone-400 text-sm mt-1 max-w-xs">${subtitle}</p>` : ''}
    </div>
  `;
}
```

This is deliberately simple — a centered icon + text block. Every view calls it when their list array is empty.

**Why a shared module?** Phase 2-4 views each had inline empty messages (or none). Centralizing means one place to restyle empty states, and junior devs call one function instead of writing HTML strings.
</action>

- Returns HTML with Material Symbol icon, title, and subtitle

## Task 2: Create `assets/js/ui/skeleton.js`

<read_first>
- assets/js/views/dashboard.js (for existing renderSkeleton pattern)
</read_first>

<action>
Create `assets/js/ui/skeleton.js`:

```js
// assets/js/ui/skeleton.js
// Reusable skeleton loading components.
// Dashboard already has a custom skeleton; this module provides
// reusable table-row skeletons for CRUD list views.

/**
 * Returns HTML for N skeleton table rows.
 * @param {number} [count=5] — Number of skeleton rows
 * @returns {string} HTML string of <tr> elements with animated bars
 */
export function skeletonRows(count = 5) {
  return Array.from({ length: count }, () => `
    <tr class="animate-pulse">
      <td class="px-4 py-3"><div class="h-4 bg-stone-200 rounded w-3/4"></div></td>
      <td class="px-4 py-3"><div class="h-4 bg-stone-200 rounded w-1/2"></div></td>
      <td class="px-4 py-3"><div class="h-4 bg-stone-200 rounded w-1/3"></div></td>
      <td class="px-4 py-3"><div class="h-4 bg-stone-200 rounded w-20"></div></td>
    </tr>
  `).join('');
}

/**
 * Returns an HTML string for a full-page loading skeleton.
 * Useful for views that load data before rendering anything.
 * @returns {string} HTML string
 */
export function skeletonPage() {
  return `
    <div class="p-6 space-y-6 animate-pulse">
      <div class="h-8 w-48 bg-stone-200 rounded"></div>
      <div class="bg-white rounded-xl shadow-sm p-5 space-y-4">
        ${Array.from({ length: 4 }, () => `
          <div class="flex items-center gap-4">
            <div class="w-10 h-10 bg-stone-200 rounded-lg"></div>
            <div class="flex-1 space-y-2">
              <div class="h-4 bg-stone-200 rounded w-1/3"></div>
              <div class="h-3 bg-stone-200 rounded w-1/2"></div>
            </div>
          </div>
        `).join('')}
      </div>
    </div>
  `;
}
```

Dashboard already has its own `renderSkeleton()` inline. Leave it as-is (it's card-shaped, not table-shaped). This module provides table-row skeletons for the CRUD views.
</action>

<acceptance_criteria>
- `assets/js/ui/skeleton.js` exists and exports `skeletonRows` and `skeletonPage`
- `skeletonRows(5)` returns 5 `<tr>` elements with `animate-pulse` and `bg-stone-200` bars
- `skeletonPage()` returns a full-page skeleton with header + content
- Both use `animate-pulse` for animation

<action>
For each CRUD view module, make these changes:

### A. Import the new modules

Add at the top:
```js
import { emptyState } from '../ui/empty-state.js';
import { skeletonRows } from '../ui/skeleton.js';
```

### B. Show skeleton rows while loading

Each view's `loadXxx(user)` function fetches data then renders. Before the `await api.get(...)` call, render skeleton rows into the table body:

```js
canvas.innerHTML = renderHeader() + `<tbody>${skeletonRows(5)}</tbody>`;
```

Adjust the exact pattern per view — some render the whole page structure, some just the table. The key point: show skeleton rows between the fetch call starting and data arriving.

### C. Show empty state when data is empty

After the data arrives, check if the array is empty. If so, show `emptyState()` instead of the table:

```js
if (items.length === 0) {
  canvas.innerHTML = renderHeader() + emptyState('inbox', 'No programs yet', 'Create your first program to get started.');
  return;
}
```

Each view should have a relevant icon and message:
- Programs: `school`, 'No programs yet', 'Create your first program to get started.'
- Announcements: `campaign`, 'No announcements yet', 'Create an announcement to share news.'
- Events: `event`, 'No events yet', 'Add an event to the calendar.'
- News: `newspaper`, 'No news articles yet', 'Publish your first news article.'
- Gallery: `photo_library`, 'No gallery images yet', 'Upload an image to get started.'
- Faculty: `school`, 'No faculty members yet', 'Add a faculty member to the directory.'
- Users: `group`, 'No users yet', 'Create a user account to get started.'
- Approvals: `verified`, 'No pending approvals', 'All content has been reviewed.'

### D. Add `data-id` to table rows (if not already present)

Every CRUD table row should have `data-id` on the `<tr>` so the search filter (from Plan 05-01) can find and show/hide rows. Check each view — most should already have this from Phase 2-4, but verify.

### E. Dashboard — no changes needed

The dashboard already has a skeleton and shows "No activities yet" for the activities section. Leave it as-is unless the visual QA pass (Task 5) identifies issues.
</action>

## Task 5: Add responsive refinements to app.css

<read_first>
- assets/css/app.css
- views/shell.html
</read_first>

<action>
Add responsive refinements to the end of `app.css`. These target specific mobile breakpoints that the utility classes in templates can't easily express:

```css
/* ── Responsive refinements (below md: 768px) ─────────────────── */

@media (max-width: 767px) {
  /* Full-width topbar search on mobile */
  #topbar-search {
    font-size: 0.875rem;
  }

  /* Stack quick-action buttons on small screens */
  #main-content .flex.flex-wrap.gap-3 {
    flex-direction: column;
  }

  /* Reduce card padding on mobile */
  #main-content .bg-white.rounded-xl.shadow-sm.p-5 {
    padding: 1rem;
  }

  /* Make modal full-width on mobile */
  #modal-overlay .bg-white,
  [id^="modal-"] .bg-white {
    max-width: 100%;
    margin: 0 1rem;
    border-radius: 1rem;
  }
}

/* ── Responsive refinements (below sm: 480px) ──────────────────── */

@media (max-width: 479px) {
  /* Smaller stat card text on very small screens */
  #main-content .text-2xl {
    font-size: 1.25rem;
  }

  /* Hide secondary table columns on very small screens */
  #main-content table .hide-sm {
    display: none;
  }
}
```

The `.hide-sm` class can be added to `<th>` and `<td>` elements for columns that are less important on mobile (e.g., "Created at" dates).
</action>

- Quick-action buttons stack vertically on mobile
